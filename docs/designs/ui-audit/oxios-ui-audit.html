화면 간 통합 — 반복되는 근본 패턴

개별 화면의 문제를 넘어, 20개 화면에 걸쳐 반복되는 시스템 수준의 패턴 3개. 각 패턴은 교차 증거(화면명 + file:line)와 함께.

패턴 01 · 6화면

표시↔동작 불일치 — 감시 전용 가드, 죽은 라벨, 역연결 액션

컨트롤의 가시 라벨/색이 실제 부작용과 어긋난 사례가 가장 많다. "한도"라 써놓고 차단은 못 하고(알림 전용), "권한"인데 읽기 전용, "Save"를 누르면 편집분이 버려지는 식이다.

비용 spend-limit-card.tsx:21-23 · 보안 security.tsx:53-65,108-146 · Git git.tsx:154-162,185-187 · 워크스페이스 routes/workspace/index.tsx:339-341 · 리소스 resources.tsx:80,99,116 · 지식 stores/knowledge.ts:35
수정 방향: 모든 컨트롤의 가시 라벨이 부작용과 일치하는지 전면 감사. 즉시 실행되는 파괴·복구 액션에 확인 다이얼로그. 감시 전용 한도/가드는 "알림 전용" 명시. 임계치 기반 색상(정상=중성).
패턴 02 · 5화면

아이콘·호버 전용 컨트롤 — 키보드/터치/저시력 접근 불가

카드 행의 편집/삭제 버튼이 32px 미만·반투명 아이콘 전용이거나, 핵심 정보가 hover 툴팁에만 있다. 키보드·터치·저시력 사용자가 액션을 발견하거나 정보를 읽지 못한다.

프로젝트 project-card.tsx:40-59 · 마운트 routes/mounts/index.tsx:113-141 · 크론 cron-timeline-view.tsx:176-205 · 지식 editor-toolbar.tsx:63-194 · 채팅 chat-input.tsx:296-313
수정 방향: 아이콘 전용 버튼에 보조 텍스트 라벨 또는 명확한 aria-label + ≥44px 타겟. hover 전용 정보는 포커스·터치로도 도달 가능하게.
패턴 03 · 5화면

탭/뷰 상태가 URL에 없고, 모드 전환이 기능 접근을 강제한다

탭 상태가 로컬 state로만 있어 새로고침·공유·뒤로가기가 끔기고, 한 뷰(캔버스/타임라인)에서만 접근 가능한 기능이 있어 모드 전환 없이는 작업을 마칠 수 없다.

수정 방향: 탭/뷰 상태를 URL search param으로 동기화. 한 뷰에서만 가능한 기능은 양 뷰에서 공통 노출 또는 뷰 전환 없이 도달 가능하게.

발견 (Findings)

  • MED

    CPU·RAM KPI가 실제 값과 무관하게 항상 경고/오류 색

    CPU 카드는 warning(호박색), RAM 카드는 error(빨강) 색으로 고정되어, 34% CPU·48% RAM 같은 정상 수치도 마치 문제인 것처럼 보인다. 색이 의미를 전달하지 못하고 거짓 경보를 만든다.

    routes/index.tsx:139 (iconClassName text-warning) · :151 (text-error) · stat-card.tsx:139-143

  • MED

    KPI 카드의 클릭 가능 여부가 불일치

    CPU·RAM·메모리 카드는 href로 연관 화면으로 이동하지만, 토큰/분·대기 승인 카드는 정적이다. 같은 그리드 안에서 어떤 카드는 눌리고 어떤 카드는 안 눌리는 것을 시각으로 구분할 수 없어 예측성이 떨어진다.

    routes/index.tsx:143,155,163 (href 부여) vs :124-132,165-174 (href 없음)

  • MED

    에이전트 OS의 홈에서 에이전트를 시작할 수 없다

    대시보드 전체가 모니터링 전용이다. 가장 중요한 액션(새 에이전트 시작 · 빠른 질문)이 페이지 어디에도 없고 사이드바/헤더로만 접근된다. "무엇을 해야 하는가"에 대한 시그널이 부재하다.

    routes/index.tsx:93-201 (CTA 없음) · header.tsx:37-64 (⌘J/⌘K는 헤더에만)

  • LOW

    대기 승인이 KPI 카드와 전체 너비 섹션에 이중 노출

    승인 건수가 KPI 카드(카운트)로 표시되는 동시에, 1건 이상이면 페이지 하단에 ApprovalsQueue가 또 나타난다. 같은 정보가 두 자리를 차지해 세로 밀도가 커진다.

    routes/index.tsx:165-174 (KPI) · :200 (조건부 ApprovalsQueue)

  • LOW

    xl(1280px)에서 KPI 6열은 카드당 ~180px로 라벨이 빡빡하다

    콘텐츠 너비(사이드바 제외) 약 1080px를 6으로 나누면 카드당 ~180px. 대문자 라벨이 자주 잘리며, sm(640px)에서 xl로 한 번에 3→6으로 건너뛰어 lg 구간의 밀도가 비효율적이다.

    routes/index.tsx:116 (grid-cols-2 sm:grid-cols-3 xl:grid-cols-6)

변경 제안

  • KPI 색상을 임계치 기반으로: 정상=중성(info), 80%↑=warning, 95%↑=error. StatCard에 threshold prop 추가.
  • 모든 KPI 카드에 연관 화면 링크 부여(토큰→비용, 승인→승인 큐). 링크 카드는 hover·화살표 affordance로 비-링크와 구분.
  • 헤더 또는 타이틀 행에 "새 에이전트" / "빠른 질문 ⌘J" 1차 CTA 배치.
  • KPI 6열을 xl→4열, 2xl→6열로 조정해 라벨 여백 확보.

  • MED

    뷰 간 기능 비대칭 — 상태 탭 필터가 테이블 뷰에만 존재

    통계 표시가 뷰마다 두 가지 형태로 갈라진다

    테이블 뷰는 StatsBar 컴포넌트로, 캔버스 뷰는 인라인 텍스트(3 running · $cost · tokens)로 통계를 보여준다. 동일 정보가 두 형태로 제공되어 일관성이 떨어진다.

  • MED

    대시보드와 마찬가지로 에이전트 목록 화면에도 "새 에이전트 시작" CTA가 없다. Agent OS의 핵심 액션이 사이드바/헤더로만 접근된다. 빈 캔버스는 특히 행동 유도가 부재하다.

    routes/agents/index.tsx:307-516 (CTA 없음)

  • LOW

    페이지네이션 버튼이 영문 하드코딩

    "← Previous" / "Next →" 가 t() 없이 문자열 리터럴로 박혀 있다. 한국어 UI에서도 영어로 표시된다.

    routes/agents/index.tsx:194 (view ?? 'canvas') · :380-391

  • MED

    활성 페르소나를 비활성화할 인-카드 방법이 없다 (단방향 토글)

    routes/personas.tsx:228-237 (활성화 버튼, enabled일 때 숨김)

  • MED

    "활성" 상태가 별 아이콘(filled, 경고색)에만 의존

    활성 상태가 색이 채워진 작은 별 하나로만 표시된다. "활성"이라는 텍스트 라벨이 없어 색맹·초기 사용자가 상태를 즉시 파악하기 어렵다.

    routes/personas.tsx:213 (Star className="text-warning fill-warning")

  • LOW

    단일 활성 vs 다중 활성 규칙이 UI에 설명 없음

    여러 페르소나를 동시에 켤 수 있는지, 한 번에 하나인지가 UI 어디에도 표시되지 않는다. 활성화 동작의 결과를 예측하기 어렵다.

    routes/personas.tsx:228-237 (활성화 로직에 안내 없음)

# Oxios 가이드
- 에이전트 시작 (바이라인 강조)
- 페르소나 관리
- 스킬 설치
선택: README.md · 보기 모드 변경 시 자동 저장 인디케이터

발견 (Findings)

  • HIGH

    툴바 "Save"가 편집 내용을 버리고 원본을 그대로 저장 — 라벨이 실제 동작과 반대

    편집 모드 진입은 파일 가져오기(useQuery ['workspace-file', path], 95-105)가 끝나야 가능하고, FileEditor에는 onChange prop이 전달되지 않는다(357-361). 툴바의 Save 버튼은 handleSave(fileData)(339-341)를 호출해 리패치된 원본을 그대로 saveFile.mutate에 넘긴다. 인-에디터 ⌘S(file-editor.tsx:52-61,37-42)는 에디터의 라이브 doc를 저장하므로 두 경로의 결과가 다르다. 사용자가 "Save"를 누르면 방금 한 편집이 무시되고 마지막 서버 상태가 다시 기록된다.

    routes/workspace/index.tsx:339-341 (handleSave(fileData)) · :357-361 (FileEditor onChange 미전달) · :95-105 (fileData query) · components/workspace/file-editor.tsx:52-61, :37-42 (Ctrl+S 경로)

  • HIGH

    삭제가 네이티브 confirm()으로만 보호됨 — 동시에 페이지 내 유일한 모달 흐름

    삭제 핸들러는 confirm(`Delete ${path}?`)로 OS 다이얼로그를 띄우고 결과에 따라 진행한다(169-178). 화면 나머지 흐름이 전부 인라인(트리 토글, 편집 모드 전환, 업로드 토글)이라 삭제 보호 방식만 시스템 톤에서 벗어난다. 프로젝트 컨텍스트 분리·실수 취소가 없고, 같은 줄에서 다른 작업으로 복귀하기 어렵다.

    routes/workspace/index.tsx:171 (confirm) · :169-178 (handleDelete)

  • MED

    보기/편집/삭제 툴바 버튼이 아이콘만, aria-label 없음 — 크론 잡과 비대칭

    툴바의 ·· 버튼(308-333)은 aria-label·툴팁·텍스트 라벨이 전혀 없고 아이콘만으로 상태를 표현한다. 동일 라우트의 다른 액션 그룹(예: cron-jobs.tsx:195-217)은 aria-label을 항상 부여한다. 스크린리더 사용자는 모드를 알 수 없고, 마우스 사용자도 hover 전에는 무엇이 active인지 식별 불가.

    routes/workspace/index.tsx:308-333 (icon-only 버튼) · 비교: routes/cron-jobs.tsx:195-217 (aria-label 일관)

  • MED

    "더블클릭으로 편집" 안내가 빈 카드 푸터에만 있고, 툴바와의 진입 경로가 둘로 갈라짐

    편집 모드 진입은 (a) 빈 상태 카드의 "더블클릭으로 편집" 안내(379-381)와 (b) 툴바의 ✎ 버튼(316-323) 두 곳에서 일어난다(handleDoubleClick는 136-142, 라우트 자체 핸들러). 비어있지 않을 때는 (a)가 보이지 않으므로 사용자는 ✎ 버튼을 발견해야 하고, 더블클릭 제스처 자체는 어디에도 설명되지 않는다.

    routes/workspace/index.tsx:136-142 (handleDoubleClick) · :316-323 (toolbar edit) · :378-381 (empty-state hint)

  • MED

    생성·업로드 1차 액션이 헤더에서 outline 버튼으로 묻히고, 업로드는 트리 푸터에 ghost 버튼으로 존재

    헤더의 "+ 생성" 버튼(250-252)은 variant="outline"으로 다른 outline 버튼과 톤이 같아 2차 액션처럼 보이고, 업로드는 트리 카드 내부의 ghost 토글(Upload file / Hide upload, 286-295)에만 존재한다. 가장 자주 쓰는 두 신규 파일 작업이 시각 무게 0인 자리에 있어, 페이지에서 신규 작성 의도가 거의 읽히지 않는다.

    routes/workspace/index.tsx:250-256 (헤더 생성·새로고침) · :286-295 (업로드 토글)

  • LOW

    트리 노드는 크기만 표시 — 파일 종류/언어 식별 불가, 미리보기 가능 여부 예측 불가

    트리 노드는 단일 File 아이콘과 크기만 보여주고(215-225), 확장자별 분기가 없다. isEditable/isImage(types/workspace.ts)처럼 미리보기 가능 여부가 미리 결정돼 있는데도 트리에서는 이를 사용자에게 알리지 않는다. 결과적으로 텍스트/이미지/JSON/마크다운 구분이 한눈에 되지 않는다.

    routes/workspace/index.tsx:215-225 (size만 표시) · types/workspace.ts (isEditable/isImage)

변경 제안

  • (HIGH) FileEditor에 onChange를 라우트에서 받아 워크스페이스 페이지 내부 버퍼로 유지하고, 툴바 Save는 그 버퍼를 저장하도록 변경. handleSave(fileData)(339-341) 제거. 또는 툴바 Save를 제거하고 ⌘S만 살아남도록 단순화.
  • 삭제 시 인라인 2단계 확인(파일 경로 + 보조 셀렉터)으로 교체. 네이티브 confirm() 제거.
  • 툴바 보기/편집/삭제 버튼에 aria-label + 텍스트 라벨(예: "보기", "편집", "삭제") 추가. 현재 모드는 버튼 자체에 visible 상태(배지 또는 active 색상)로 표시.
  • 빈 상태 카드의 "더블클릭으로 편집" 안내는 제거하고, 툴바 또는 편집기 자체 진입 시 한 줄 짧은 안내만 유지.
  • 생성·업로드를 헤더 1차 액션(Primary 버튼)으로 승격. 트리 푸터의 업로드 토글은 보조로 강등(또는 제거하고 헤더에만 두기).
  • 트리 노드에 확장자별 아이콘 분기(JS/MD/JSON/이미지) 적용.
