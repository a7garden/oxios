발견 (Findings)

  • MED

    헤더가 비어 있다 — 오른쪽 액션 자리가 공백

    페이지 헤더의 오른쪽(justify-between의 두 번째 자식)이 아무것도 렌더하지 않는다. 페르소나·스킬 화면은 이 자리에 생성/새로고침 버튼을 둔다. 토큰 맥싱은 헤더가 텅 빈 채로 균형이 맞지 않는다.

    routes/token-maxing.tsx:20-25 (헤더, 오른쪽 자식 없음)

  • MED

    화면 목적이 한눈에 들어오지 않는다

    "토큰 맥싱"이라는 이름과 한 줄 부제만으로, 이 화면이 남은 토큰 할당량을 소진해 에이전트를 최대한 구동하는 기능이라는 것을 파악하기 어렵다. RFC-031 컨텍스트가 UI에 드러나지 않는다.

    routes/token-maxing.tsx:12-23 (부제만, 목적 설명 없음)

  • LOW

    시작 흐름이 두 가지 경로(window / 수동) + metered 확인 다이얼로그로 복잡

    윈도우 시작과 수동 시작 두 변형이 있고, metered 과금 제공자가 있으면 추가 확인 다이얼로그가 끼어든다. 첫 사용자가 어떤 버튼을 눌러야 할지 결정하기 어렵다.

    components/token-maxing/controls.tsx:50-51,59-77 (metered guard + 두 시작 변형)

  • LOW

    상태 카드에 정보 밀도가 높다

    상태 카드 하나에 running/idle 배지, 제공자+태스크, 세션 토큰 합계, 활성 윈도우(또는 수동)가 모두 들어간다. 한 카드에 4개 필드 영역이 밀집되어 있다.

    components/token-maxing/status-header.tsx:63-100 (Field/WindowField 다수)

변경 제안

  • 헤더 오른쪽에 오늘 사용량 칩 또는 새로고침/도움말 버튼 추가.
  • 부제에 한 줄 목적 설명 보강 ("남은 할당량을 자동 소진해 에이전트를 최대 구동").
  • 시작 버튼에 tooltip/도움말로 window vs 수동 차이 안내.

발견 (Findings)

  • MED

    탭(서버/도구/테스트) 상태가 uncontrolled defaultValue라 URL에 없다

    Tabs defaultValue="servers"로 탭이 제어되지 않는다. 새로고침하면 항상 서버 탭으로 리셋되고, 특정 탭으로 직접 링크하거나 공유할 수 없다.

    routes/mcp.tsx:33 (defaultValue="servers", 제어형 아님)

  • MED

    헤더의 "서버 추가"가 도구/테스트 탭에서도 보인다

    "서버 추가" 버튼은 헤더에 고정되어 모든 탭에서 보이지만, 실제로 영향을 주는 건 서버 탭뿐이다. 도구·테스트 탭에서는 문맥에 맞지 않는 액션이다.

    routes/mcp.tsx:28-30 (헤더 버튼) · :33-69 (탭과 무관)

  • LOW

    연결 오류 서버의 복구 액션이 행 수준에 즉시 보이지 않는다

    오류 상태 서버(github)가 있을 때 재연결/진단 액션이 행에 즉시 노출되는지가 ServerList 구현에 의존한다. 오류를 한눈에 보고 바로 조치하기엔 헤더 요약이 부족하다.

    routes/mcp.tsx:46-52 (ServerList에 위임) · components/mcp/server-list.tsx

  • LOW

    탭 카운트(서버/도구 수)가 탭 라벨에 없다

    서버 탭과 도구 탭에 각각 몇 개가 있는지 카운트가 표시되지 않아, 어느 탭에 내용이 있는지 라벨만으로 파악해야 한다.

    routes/mcp.tsx:35-43 (TabsTrigger에 카운트 없음)

변경 제안

  • 탭을 제어형(value/onValueChange)으로 + URL search 동기화.
  • "서버 추가" 버튼은 서버 탭에서만, 또는 탭별로 문맥 액션 분리.
  • 각 탭에 카운트 배지 추가; 오류 서버는 헤더 요약에 노출 + 행에 재연결 버튼.

  • MED

    탭(개요/설정/이력/템플릿) 상태가 URL에 동기화되지 않는다

    activeTab이 로컬 useState로만 관리된다. 새로고침·공유·직접 링크로 특정 탭을 열 수 없다.

    routes/email.tsx:158 (useState) · :201 (Tabs value)

  • MED

    configured 사용자가 설정 탭을 열면 개요로 강제 이동한다

    상태가 로드되어 configured가 참이면 setActiveTab으로 setup→overview로 옮긴다. 사용자가 의도적으로 재설정 탭을 열려 해도 예고 없이 이동해 예측 가능성을 해친다.

    routes/email.tsx:164-166 (configured 시 setup→overview 강제)

  • LOW

    4개 탭이 configured 여부와 무관하게 모두 표시

    미설정 상태에서도 발송 이력·템플릿 탭이 보인다. 설정 전에는 의미 없는 탭이 노출되어 첫 진입 인지 부하가 있다.

    routes/email.tsx:201-219 (항상 4개 TabsTrigger)

  • LOW

    이력·템플릿 탭에 건수 카운트가 없다

    발송 이력·템플릿 탭 라벨에 각각 몇 건인지 카운트가 없다. 개요 카드의 총 발송 수에서만 유추할 수 있다.

    routes/email.tsx:211-218 (TabsTrigger에 카운트 없음)

변경 제안

  • 탭을 URL search(?tab=)와 동기화 — 새로고침·공유·히스토리 지원.
  • configured 사용자의 setup 탭 진입은 강제 이동 대신 배너로 "이미 구성됨" 안내.
  • 미설정 시엔 setup 중심으로 탭 축소; 이력/템플릿 탭에 카운트 배지.
