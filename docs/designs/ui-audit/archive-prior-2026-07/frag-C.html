# Oxios 가이드
- 에이전트 시작 (바이라인 강조)
- 페르소나 관리
- 스킬 설치
선택: README.md · 보기 모드 변경 시 자동 저장 인디케이터

발견 (Findings)

  • HIGH

    툴바 "Save"가 편집 내용을 버리고 원본을 그대로 저장 — 라벨이 실제 동작과 반대

    편집 모드 진입은 파일 가져오기(useQuery ['workspace-file', path], 95-105)가 끝나야 가능하고, FileEditor에는 onChange prop이 전달되지 않는다(357-361). 툴바의 Save 버튼은 handleSave(fileData)(339-341)를 호출해 리패치된 원본을 그대로 saveFile.mutate에 넘긴다. 인-에디터 ⌘S(file-editor.tsx:52-61,37-42)는 에디터의 라이브 doc를 저장하므로 두 경로의 결과가 다르다. 사용자가 "Save"를 누르면 방금 한 편집이 무시되고 마지막 서버 상태가 다시 기록된다.

    routes/workspace/index.tsx:339-341 (handleSave(fileData)) · :357-361 (FileEditor onChange 미전달) · :95-105 (fileData query) · components/workspace/file-editor.tsx:52-61, :37-42 (Ctrl+S 경로)

  • HIGH

    삭제가 네이티브 confirm()으로만 보호됨 — 동시에 페이지 내 유일한 모달 흐름

    삭제 핸들러는 confirm(`Delete ${path}?`)로 OS 다이얼로그를 띄우고 결과에 따라 진행한다(169-178). 화면 나머지 흐름이 전부 인라인(트리 토글, 편집 모드 전환, 업로드 토글)이라 삭제 보호 방식만 시스템 톤에서 벗어난다. 프로젝트 컨텍스트 분리·실수 취소가 없고, 같은 줄에서 다른 작업으로 복귀하기 어렵다.

    routes/workspace/index.tsx:171 (confirm) · :169-178 (handleDelete)

  • MED

    보기/편집/삭제 툴바 버튼이 아이콘만, aria-label 없음 — 크론 잡과 비대칭

    툴바의 ·· 버튼(308-333)은 aria-label·툴팁·텍스트 라벨이 전혀 없고 아이콘만으로 상태를 표현한다. 동일 라우트의 다른 액션 그룹(예: cron-jobs.tsx:195-217)은 aria-label을 항상 부여한다. 스크린리더 사용자는 모드를 알 수 없고, 마우스 사용자도 hover 전에는 무엇이 active인지 식별 불가.

    routes/workspace/index.tsx:308-333 (icon-only 버튼) · 비교: routes/cron-jobs.tsx:195-217 (aria-label 일관)

  • MED

    "더블클릭으로 편집" 안내가 빈 카드 푸터에만 있고, 툴바와의 진입 경로가 둘로 갈라짐

    편집 모드 진입은 (a) 빈 상태 카드의 "더블클릭으로 편집" 안내(379-381)와 (b) 툴바의 ✎ 버튼(316-323) 두 곳에서 일어난다(handleDoubleClick는 136-142, 라우트 자체 핸들러). 비어있지 않을 때는 (a)가 보이지 않으므로 사용자는 ✎ 버튼을 발견해야 하고, 더블클릭 제스처 자체는 어디에도 설명되지 않는다.

    routes/workspace/index.tsx:136-142 (handleDoubleClick) · :316-323 (toolbar edit) · :378-381 (empty-state hint)

  • MED

    생성·업로드 1차 액션이 헤더에서 outline 버튼으로 묻히고, 업로드는 트리 푸터에 ghost 버튼으로 존재

    헤더의 "+ 생성" 버튼(250-252)은 variant="outline"으로 다른 outline 버튼과 톤이 같아 2차 액션처럼 보이고, 업로드는 트리 카드 내부의 ghost 토글(Upload file / Hide upload, 286-295)에만 존재한다. 가장 자주 쓰는 두 신규 파일 작업이 시각 무게 0인 자리에 있어, 페이지에서 신규 작성 의도가 거의 읽히지 않는다.

    routes/workspace/index.tsx:250-256 (헤더 생성·새로고침) · :286-295 (업로드 토글)

  • LOW

    트리 노드는 크기만 표시 — 파일 종류/언어 식별 불가, 미리보기 가능 여부 예측 불가

    트리 노드는 단일 File 아이콘과 크기만 보여주고(215-225), 확장자별 분기가 없다. isEditable/isImage(types/workspace.ts)처럼 미리보기 가능 여부가 미리 결정돼 있는데도 트리에서는 이를 사용자에게 알리지 않는다. 결과적으로 텍스트/이미지/JSON/마크다운 구분이 한눈에 되지 않는다.

    routes/workspace/index.tsx:215-225 (size만 표시) · types/workspace.ts (isEditable/isImage)

변경 제안

  • (HIGH) FileEditor에 onChange를 라우트에서 받아 워크스페이스 페이지 내부 버퍼로 유지하고, 툴바 Save는 그 버퍼를 저장하도록 변경. handleSave(fileData)(339-341) 제거. 또는 툴바 Save를 제거하고 ⌘S만 살아남도록 단순화.
  • 삭제 시 인라인 2단계 확인(파일 경로 + 보조 셀렉터)으로 교체. 네이티브 confirm() 제거.
  • 툴바 보기/편집/삭제 버튼에 aria-label + 텍스트 라벨(예: "보기", "편집", "삭제") 추가. 현재 모드는 버튼 자체에 visible 상태(배지 또는 active 색상)로 표시.
  • 빈 상태 카드의 "더블클릭으로 편집" 안내는 제거하고, 툴바 또는 편집기 자체 진입 시 한 줄 짧은 안내만 유지.
  • 생성·업로드를 헤더 1차 액션(Primary 버튼)으로 승격. 트리 푸터의 업로드 토글은 보조로 강등(또는 제거하고 헤더에만 두기).
  • 트리 노드에 확장자별 아이콘 분기(JS/MD/JSON/이미지) 적용.

발견 (Findings)

  • HIGH

    타임라인 뷰는 잡별 편집·활성 토글·삭제가 도달 불가 — 모드 전환이 강제됨

    기본 진입이 타임라인(viewMode='timeline', 32)이고, CronTimelineView의 각 행(113-209)은 fire marker와 스케줄 라벨만 렌더하고 row-level 액션 버튼을 갖지 않는다. 반면 목록 뷰의 카드(165-223)는 잡마다 편집(Pencil)/활성 토글(Power/PowerOff)/삭제(Trash2)를 가진다(aria-label 포함). 사용자가 타임라인에서 잡을 더블클릭하거나 행을 선택해 액션을 기대해도 보이는 경로가 없다.

    routes/cron-jobs.tsx:32 (기본 timeline) · cron-timeline-view.tsx:113-209 (row 렌더) · routes/cron-jobs.tsx:194-219 (목록 액션 비교)

  • HIGH

    타임라인 fire marker가 hover 전용 툴팁 — 키보드/터치 접근 불가

    각 marker의 이벤트 핸들러는 onMouseEnter/onMouseLeave(cron-timeline-view.tsx:194-202)뿐이고 role/tabIndex가 부여되지 않는다. 시각적 "next run" 표식(ft.isPrimary, 182-191)과 일반 fire marker의 구분을 키보드가 인지하지 못한다. 터치 디바이스에서는 첫 탭이 marker 식별을 야기한다.

    cron-timeline-view.tsx:176-205 (marker 핸들러) · :214-238 (fixed 툴팁)

  • MED

    기본 진입이 타임라인 — 신규 사용자에게 학습 비용 화면이 첫 인상

    viewMode 초기값이 'timeline'(32)이며 페이지 헤더의 토글이 "타임라인"을 active로 표시한다(307-308). 도메인 지식이 필요한 시각화(isPrimary vs 일반 marker, 시간대 그라데이션)를 첫 화면에 노출한다.

    routes/cron-jobs.tsx:32 (초기값) · :307-310 (토글 표시)

  • MED

    비활성 잡 표현이 타임라인은 시각적으로, 목록은 배지만으로 — 일관성 깨짐

    타임라인은 비활성 행에서 라벨을 muted, row 전체 opacity: 0.55, 마커까지 불투명도 0.3으로 표면 차원에서 약화시킨다(cron-timeline-view.tsx:129-130, 187, 334). 목록 카드는 Badge variant="secondary"(cron-jobs.tsx:172-174)만으로 같은 상태를 표시하고 row 자체는 정상 톤을 유지한다. 두 뷰가 같은 정보를 다르게 그려 일관성이 깨진다.

    cron-timeline-view.tsx:129-130 (row muted) · routes/cron-jobs.tsx:172-174 (목록 배지만)

  • MED

    "생성"·"편집" 두 폼이 비대칭 — 인라인 Card vs Dialog, 같은 엔티티에 두 형제 폼

    생성 폼은 페이지 안에 인라인 <Card>(123-154)로 펼쳐지고, 편집은 모달 <EditCronDialog>(225-238)로 열린다. 같은 입력 필드(name / schedule / goal)·같은 검증(!name || !schedule || !goal, 144)이 두 곳에 분기되어 있어 사용자가 편집을 시도할 때 갑자기 모달로 인터페이스가 바뀐다.

    routes/cron-jobs.tsx:123-154 (인라인 폼) · :225-238 (EditCronDialog) · edit-cron-dialog.tsx 전체

  • LOW

    "목표" 입력에 라벨 없고 i18n 키 누락 — 영문 fallback 노출 가능

    인라인 생성 폼의 <Input>(135-139)은 placeholder={t('cronJobs.goalPlaceholder', 'Goal or prompt for the agent')}만 받고, 둘러싼 <Label>/<label>가 없다. goalPlaceholder는 ko/en i18n JSON에 정의돼 있지 않으므로 두 번째 인자(영문)가 그대로 화면에 노출된다. (참고: 같은 폼의 CronScheduleEditor는 내부적으로 <Label>를 사용한다 — cron-schedule-editor.tsx:186.)

    routes/cron-jobs.tsx:135-139 (goal Input) · i18n/locales/ko.json, en.json (goalPlaceholder 미정의) · cron-schedule-editor.tsx:186 (label 사용)

변경 제안

  • (HIGH) 타임라인 행에 작업 액션을 인라인으로 노출(우측에 ✎/⏼/🗑 아이콘 또는 호버 시 패널 슬라이드). 또는 각 행이 상세 사이드 패널을 열어잡 액션을 일관되게 보여주도록 라우트 위에서 처리.
  • (HIGH) fire marker에 role="button" + tabIndex={0} 부여, onKeyDown(Enter/Space)로도 툴팁 활성화. 모바일은 long-press로 대체.
  • 기본 view를 'list'로 변경, 헤더 토글을 "고급" 톤으로 강등. 첫 사용자 가벼움 우선.
  • 목록 카드의 비활성 행을 타임라인과 동일한 muted + opacity 0.55 룰로 통일. 토글 상태는 텍스트 라벨·색상으로 명시.
  • 생성 폼을 EditCronDialog와 동일한 Dialog로 통일. 인라인 Card 제거 또는 "고급" 섹션으로 강등.
  • i18n JSON에 cronJobs.goalPlaceholder(ko/en) 추가, <Input> 위에 <Label> 부착.

발견 (Findings)

  • HIGH

    "지출 한도"가 모니터링 알림 전용 — 라벨/힌트는 차단처럼 읽힘

    JSDoc은 명시적으로 "Phase 1: monitoring + visual alert only. Phase 2 will add pre-execution enforcement"라고 선언한다(spend-limit-card.tsx:21-22). 화면의 한도 카드는 진도 바·경고 배지·"한도 초과/임박" 메시지로 마치 임계치 도달 시 실행이 멈추는 듯 표시하지만, 실제로는 알림만 뜬다. 사용자는 추가 비용이 차단된다고 오해한 채 안심한다.

    components/cost/spend-limit-card.tsx:21-23 (JSDoc), :32-50 (isOver/isNear/badge) · i18n cost.spendLimitHint · routes/budget.tsx:40 (SpendLimitCard)

  • HIGH

    같은 라우트에 두 종류의 한도가 섞여 있고 헤더는 한 쪽만 표시

    라우트 상단에는 USD 기반 SpendLimitCard가, 페이지 하단에는 토큰/호출 기반 BudgetManagement가 등장한다. 페이지를 나누지 않고 같은 스크롤 위에 늘어놓았으나 두 개념의 차이가 헤더·중간 헤딩(h2 budget.title)·컴포넌트 이름 어디에서도 설명되지 않는다. 사용자는 "예산(= 한도)"으로 통합 인식한다.

    routes/budget.tsx:40 (SpendLimitCard) · :71 (h2 budget.title) · :72 (BudgetManagement) · cost/budget의 컴포넌트 분기

  • MED

    상단 기간 탭이 화면 일부에만 적용 — 탭과 데이터의 관계가 깨짐

    Tabs(PERIODS: 오늘/7일/30일/전체)는 CostSummaryCards, CostByModel, CostByProject에만 period prop으로 전달된다(budget.tsx:42-63). SpendLimitCard는 자체 /api/costs/spend-limit만, ProviderQuotaCards는 quota API, CostChart는 30일 고정(days={30}, cost-chart.tsx:11-12), BudgetManagement는 기간 인자 자체가 없다. "30일" 선택해도 차트는 변하지 않고, "오늘" 선택해도 한도·프로바이더 카드는 변하지 않는다.

    routes/budget.tsx:42-63 (탭) · :40 (SpendLimitCard 미수신) · :58 (CostChart days 고정) · :67 (ProviderQuotaCards) · :72 (BudgetManagement)

  • MED

    에이전트 카드의 3개 액션(편집/초기화/제거) 모두 아이콘만 — 파괴/비파괴 구분 없음

    AgentBudgetCard(agent-budget-card.tsx:117-137)는 "편집/초기화/제거"가 모두 같은 outline ghost 버튼이며, 색 구분(파괴=error 색) 외엔 위치·스타일이 동일하다. resetWindow는 윈도우 리셋이라 사실상 비파괴지만 한 번에 3개 액션이 늘어서면 사용자는 무엇이 무엇을 트리거하는지 학습이 필요하다.

    components/budget/agent-budget-card.tsx:117-137 · set-budget-dialog.tsx 전체 (3-dialog 모달도 별도)

  • MED

    URL은 /budget인데 페이지 타이틀은 cost.title("비용 및 지출")

    라우트는 createFileRoute('/budget')(budget.tsx:16)로 등록되지만 헤더는 t('cost.title')("비용 및 지출", i18n/ko.json:1310)을 쓴다. 사이드바 항목 이름("비용", i18n:67)과 라우트 경로(/budget)가 한 단어 체계로 어긋난다. URL 공유/딥링크에서 의도가 흐려진다.

    routes/budget.tsx:16 (route='/budget') · :30 (t('cost.title')) · i18n/locales/ko.json:1310, 67

  • LOW

    모니터링/예산 두 영역의 progress bar 색 임계값이 일치하지 않음

    SpendLimitCard는 80%에서 warning, 100%에서 error(spend-limit-card.tsx:32-33). AgentBudgetCard는 70%에서 warning, 90%에서 error(agent-budget-card.tsx:88, 104). 같은 의미(임박/소진)를 다른 임계에서 발화시켜 사용자가 한 페이지 안에서 두 영역을 동시에 읽을 때 느끼는 위기감의 강도가 불일치한다.

    components/cost/spend-limit-card.tsx:32-33, :62-67 · components/budget/agent-budget-card.tsx:88, 104

변경 제안

  • SpendLimitCard 상단에 "모니터링 전용" 배너를 영구 노출. i18n cost.spendLimitHint를 "강제 차단은 향후 단계에서 제공됩니다" 문구로 교체. 색은 항상 warning 유지.
  • /budget 단일 라우트를 두 서브섹션(/budget/spend, /budget/agents) 또는 명확한 두 영역 헤딩("지출 한도(USD)" / "에이전트 예산(토큰·호출)")으로 분리. 헤딩에 의미 명시.
  • 기간 탭을 페이지 전역에 적용하거나, 적용 범위를 탭 근처에 명시("기간 적용: 차트·모델별"). CostChart는 탭과 동기화.
  • 에이전트 카드의 액션에 라벨 부착. "제거"만 error 색상. 1차 액션은 편집, 2차는 초기화, 3차(파괴)는 분리 영역.
  • 라우트 path를 /cost로 옮기거나 사이드바 라벨과 페이지 타이틀을 통일. i18n 정합성 회귀 테스트.
  • 두 progress bar의 임계치 룰을 공통 함수 thresholdLevel(pct)로 통일. 설정 화면에서 한 곳에서 조정 가능하도록.
