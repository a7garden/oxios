발견 (Findings)

  • HIGH

    @ 컨텍스트 검색이 두 종류의 popover로 분리되어 사용자에게 보이지 않는다

    chat-input은 @ 입력 시 knowledge + memory + mount를 통합 검색하는 popover를 띄우지만, 슬래시 /는 knowledge-chat(다른 화면)에만 routes 메뉴를 띄운다. 같은 입력 박스에서 두 가지 "주제 트리거"가 다르게 동작하는데 UI에는 시그널이 없어, 처음 사용자는 @ vs /가 컨텍스트 첨부 vs 라우팅이라는 사실을 알 수 없다.

    routes/chat.tsx:24-272 · components/chat/chat-input.tsx:296-313 (detect /@) · components/chat/chat-input.tsx:330-367 (popover render) · components/knowledge/knowledge-chat.tsx:278-282 (slash in 별도 화면)

  • HIGH

    활동 타임라인이 기본 접혀 있어 투명성이 무력화된다

    RFC-015의 핵심 약속은 "추론·도구 호출을 사용자에게 실시간으로" 노출하는 것이지만, ActivityTimeline은 timelineCollapsed = activities.length > 8 외에는 카운터만 보이는 헤더만 노출하고 본문 카드는 클릭해야 펼쳐진다. 위 데모처럼 reasoning 1 + tool 2건 정도면 항상 한 번 클릭이 필요해, 화면 상단 활동이 "잠긴" 것처럼 느껴진다.

    components/chat/activity-timeline.tsx:28 (collapseAfter=8) · :32 (timelineCollapsed 초기값) · :42-95 (헤더만)

  • MED

    live-activity-bar / activity-timeline / metadata-footer가 같은 정보를 중복 표시한다

    스트리밍 중에는 LiveActivityBar가 "Thinking… / Running X" 한 줄을, 답변 시작 후엔 ActivityTimeline이 같은 도구 호출을 카드로, 푸터는 다시 "in/out tok · duration_ms · eval ✓"을 노출한다. 활동 카드 헤더는 "2 tool calls · 122k+1.9k tok"라고 집계하는데, 푸터의 in/out이 또 노출되어 동일 사실이 두 곳에 표시된다.

    components/chat/live-activity-bar.tsx:25-72 · components/chat/activity-timeline.tsx:55-72 (token count) · components/chat/message-bubble.tsx:184-193 (footer tokens)

  • MED

    추론 델타 머지가 tail-only lookup이라 중간 삽입에 깨진다

    과거 "빠른질문에서 reasoning이 토큰마다 한 블록씩" 버그 수정 후에도 mergeOrAppendActivity(stores/chat.ts:279-314)의 reasoning 분기(:298-311)는 existing[existing.length - 1] 단일 슬롯만 본다. 두 reasoning 델타 사이에 어떠한 다른 activity(tool_call·phase·memory·usage) 한 건만 끼어도, 뒤쪽 델타는 형제를 못 찾고 새 reasoning 블록을 append해 동일-소스 사고 스팬이 둘로 쪼개진다. 정상 트래픽(추론→도구→추론)에서도 재현되어 과거 버그와 같은 시각 증상이 다시 노출된다. 키 가드(reasoningSource || content.slice(0, 16) || message.id)로 existing.findIndex(...)를 쓰지 않으면 실질적으로 보호되지 않는다.

    stores/chat.ts:279-314 (mergeOrAppendActivity reasoning branch, tail-only existing[existing.length - 1]) · :298-311 (source guard) · :1212-1262 (handleChunk phase/tool_start/.../reasoning 끼어드는 경로) · 보조: _pendingTokens·_tokenRafId 싱글톤(stores/chat.ts:528-529)이 모듈 전역 mutable. 단일 스트림에선 F9 rAF 배치가 잘 작동하지만, 새 세션(newSession·loadSession·setActiveProject·removeMessage) 시점에 discardPendingTokens(:570-576)를 모두 호출하지 않으면 늦게 도착한 rAF가 다음 세션의 메시지 배열을 오염시킬 수 있다 — 이미 4개 진입점에서 호출되지만 빠뜨리면 silent corruption.

  • MED

    모델/롤 선택이 한 줄 pill에 같이 묶여 한쪽이 잘린다

    chat-input.tsx의 하단 바는 ModelPickerContainer(모델+롤 + 검색)을 flex-1 한 칸에 넣고, 우측으로 queued/Send를 몰아넣는다. 결과적으로 위 데모처럼 모델명(deepseek-v3-flash) 뒤로 롤/검색 텍스트(role: research)가 overflow:hidden;text-overflow:ellipsis로 잘린다. 사용자는 활성 모델과 라우팅 룰 중 어느 것이 적용됐는지 시각적으로 구분할 수 없다.

    components/chat/chat-input.tsx:446-491 (input bottom bar) · :447-455 (ModelPicker flex-1) · components/chat/model-picker.tsx:97-424 (combined pill)

  • MED

    프로젝트 자동감지 배너가 입력창을 위에서 덮어 세션 시작을 방해한다

    AiDetectionBadge(헤더)와 MountDetectionBadge(마운트), 그리고 !connected 재연결 배너가 모두 입력창 위쪽 스택에 쌓여 동시에 노출될 수 있다. 데모처럼 세 가지가 한 번에 뜨면 입력 박스가 화면 하단 1/3 아래로 밀려, 사용자가 "지금 보낼 수 있나"를 인지하기 어렵다. 닫기 affordance는 있지만 세션 시작 의도와 충돌할 때 우선순위가 없다.

    routes/chat.tsx:154-188 (재연결·감지 스택) · components/chat/chat-input.tsx:419-421 (비활성 input box)

  • MED

    Empty-state의 capability 그리드는 신규 사용자 온보딩에만 도움 되고 진행 세션에서는 발견되지 않는다

    EmptyChatState는 knowledge·memory·tools·projects 4개 진입점을 그리드로 노출하지만, messages.length === 0인 경우에만 렌더되고 세션이 진행되면 MessageBubble 흐름으로 즉시 사라진다. "지금 이 모델/마운트로 무엇을 할 수 있는가"는 매 세션마다 필요한 정보인데 한 번도 노출되지 않는다.

    routes/chat.tsx:198 (EmptyChatState 조건) · components/chat/empty-chat-state.tsx:78-98 (capability grid) · empty-chat-state.tsx:100-128 (Recent sessions)

  • LOW

    ⌘⇧N 단축키만 헤더에 노출되고 ⌘J·⌘K의 의미가 다르다

    Topbar는 ⌘J "Quick ask"·⌘K "검색"·⌘⇧N "새 채팅"을 같은 형식으로 노출하지만, ⌘J는 별도 QuickAskDialog를 띄우는 모달이고 ⌘K는 CommandPalette, ⌘⇧N은 라우트 변경이다. 같은 톤의 칩 세 개가 실제로는 다른 "입력 모드"를 가리킨다.

    routes/chat.tsx:90-100 (⌘⇧N 핸들러) · components/chat/chat-input.tsx:431-434 (input placeholder 분기)

변경 제안

  • 트랜스페어런시 한 단계 올리기: ActivityTimeline 기본 timelineCollapsed = false, 활동이 ≤3건일 땐 헤더 스트립 없이 본문 카드만 펼친다. 헤더 카운터는 큰 활동(8+)에서만 등장.
  • mergeOrAppendActivity의 reasoning 분기에 키 가드 추가: reasoningSource가 비어 있으면 마지막 reasoning 블록의 id 를 키로 사용, id 도 없으면 컨텐츠 slice(0, 16) 으로 fallback. 같은 id면 델타로 머지.
  • 모델 pill을 입력박스의 우측 → 별도 한 줄 위로 옮기고, 롤은 우측 보조 pill. 양쪽 모두 truncation 없이 노출.
  • 재연결·프로젝트 감지·마운트 감지 배너를 한 줄 "status bar"로 통합. 입력창 위쪽을 깨끗하게 유지.
  • Empty-state capability 그리드를 "현재 세션 컨텍스트(모델·마운트·프로젝트)" 카드 옆의 작은 패널로 항상 노출. 진행 중에는 minimized 상태로 표시.
  • ⌘J / ⌘K / ⌘⇧N topbar 버튼에 작은 모드 hint(다이얼로그 / 팔레트 / 라우트) 아이콘 추가.
  • @ mention 검색에서 잘린 mount name 처리를 ellipsis가 아닌 두 줄 표시로.
